# Features/ReactionCam/ ‚Äî driver PiP (Phase 6 ‚Äî built)

Concept: tap the CAM button on the iPad (toggle ‚Äî tap again to drop it) ‚Üí circular PiP appears on the TV showing that player's driver reacting to live physics. Dev/sim shortcut: launch arg `--show-cams` forces every PiP on.

Files (as built 2026-07-18)
- `ReactionCamView.swift` ‚Äî circular-masked mini `RealityView`: driver bust (Quaternius rig, painted by `DriverPainter` from `design.driver` ‚Äî the kid's character colors, hats and all, since C4/C5) + `PerspectiveCamera`, key + cool rim light, player-colored ring, name capsule. The bust rolls/glances into turns with the live car (`director.lean` √ó RaceTuning.reactionLeanAngle). Behind it, `ReactionBackdrop` (same file) draws radial anime speed lines whose density/length/drift track the car's real speed, tipped by the lean, over a vignette that tints warm on boost and red on crash. Boost/celebrate spring-pops the whole PiP; crash knocks it askew. Positioned bottom-left/right per player by ArenaView. Face bubble uses the character's skin tone + face paint (`driver.faceDrawingPNG`, falling back to the legacy `CarDesign.faceDrawingPNG`).
- `ReactionDirector.swift` ‚Äî race events ‚Üí reaction state machine: `idle ‚Üí steerLeft/steerRight ‚Üí braced ‚Üí boosted / crashed / celebrating`. Continuous inputs: yaw rate (lean) + loop-within-0.5 s (brace) + speed01; also exposes smoothed `lean` (‚àí1‚Ä¶1) and `speed01` readouts for the PiP (blend rate RaceTuning.reactionMotionSmoothing). Discrete events override instantly. Min state hold (RaceTuning.reactionMinHold) so it never flickers; celebrating is sticky. Unit-tested.
- `ReactionFeed.swift` ‚Äî bridges live `RaceSession` state ‚Üí one director per racer, every frame from ArenaView's scene subscription. Discrete events detected by diffing racer stats (crashes‚Üë = crashed, meter 1‚Üí0 = boosted, finishTime set = celebrating) ‚Äî no extra event plumbing.
- `DriverPoser.swift` ‚Äî `make(profile:)` builds the bust painted by `DriverPainter`, then plays the matching Quaternius clip (idle/Punch/Death/Jump ‚Üí idle/boosted/crashed/celebrating), crossfaded 0.15 s. Clips are separate USDZs (`driver-*.usdz`) from `tools/convert_driver_rig.py`; skeletons match so any clip plays on the one bust. **The bust ships on the idle rig alone and the event clips stream in behind it** (`loadEventClips`) ‚Äî awaiting all four rigged USDZs up front (they load serially on the main actor) left the PiP an empty circle for ~14 s on a cold launch. `apply` falls back to the idle clip for anything not in yet, so arriving early only costs a plain-faced boost for a moment.
- `FaceDecals.swift` ‚Äî face expression per state: hand-drawn vector paths composited over the PiP instead of the planned texture-swap quad ‚Äî reads better at PiP size, zero texture authoring, and no emoji (CLAUDE.md). Steering/braced states reuse the idle clip; the face carries them.

Live-editing the driver (C6): the character editor renders this same PiP while a kid picks hair and hats, so `ReactionCamView` swaps its bust when `design.driver` changes (`rebuildIfDriverChanged`, guarded by `DriverPainter.appearanceSignature`). It **rebuilds** rather than repaints: `DriverPainter.apply` on a bust that's already mid-animation swaps the wardrobe geometry but leaves the stripe texture stale, so a kid going bald kept their hair colour. A fresh poser is cheap ‚Äî AssetStore caches the rig, DriverPainter caches the texture. Races never hit this path; the driver can't change mid-lap.

Deviations from the original plan
- No Blender-authored seated/lean/brace poses: the rig's stock clips + face badges cover every state. In-car "seated" pose = standing rig sunk hip-deep into the chassis (legs hidden ‚Äî RaceTuning.driverSinkRatio).
- Sprite fallback not needed in Simulator; profile the second RealityView on real Apple TV hardware before shipping (fallback plan unchanged if it costs > 10% frame time).
