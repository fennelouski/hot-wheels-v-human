# Features/Arena/ — the 3D race scene (Phases 1–2 solo, 3 on TV)

Files
- `ArenaView.swift` — the `RealityView`: builds scene from `AppModel` blueprint via `TrackSpawner`, spawns cars via `CarFactory`, registers `DriveSystem`/`RaceRulesSystem`, directional light, inline chase camera (see below). Hosts overlay HUD.
- `ArenaEnvironment.swift` — the world around the track: procedural sky dome (gradient + puffy clouds on candy/day/sunset, stars on space — both drawn chunky because the chase cam only ever samples a thin band of a texture that wraps 360°; clouds are redrawn a texture-width to each side so the seam can't slice them) + 90 m checkered play-mat ground + scattered toy props (item-* USDZ), themed per trackId (candy/day/sunset/space — order is load-bearing, byte-sum % 4). Rebuilt per race via `RaceSession.trackID`/`trackFootprint`; props stay out of the track's footprint and carry no collision. Coin props spin via `SpinComponent`/`SpinSystem` (same file) at varied rates — not an `OrbitAnimation`, which orbits a centre and so does nothing at radius 0; other props stay still because a spinning traffic cone reads as a glitch.
- `ArenaHUDView.swift` — TV overlay: countdown numerals, per-player name+lives banners, lap counter, finish banner, results panel (times, crashes, best segment, REMATCH prompt "press READY on iPad").
- Chase camera — inline in `ArenaView`'s update subscription (no separate `CameraSystem.swift`): follows the cars' midpoint, pulls back proportional to their separation, smooth lerp. **Loop swing:** a loop's circle lies in the plane of travel, so from straight behind it's edge-on and reads as a wall — while a car is on or nearing a loop (detected via `LaneFollowComponent.loopRanges`, lead-in `RaceTuning.loopCamLead`) the camera eases to a 3/4 side angle (`loopBias`) so the ring reads as a circle. Finish/crash special cams are not built yet. Solo Arena on iPad reuses everything.
- `SoloArenaView.swift` — iPad wrapper: Arena + embedded mini-Dashboard side panel via `LoopbackTransport` (this is Test Mode's home too — Test Mode = Solo Arena with `MatchConfig.mode == .test`, two local designs, no lives/boost). Also home to the **workshop "try it" kit**, since every use of it is "present a Solo Arena over what I'm building":
  - `.racePreview(isPresented:designs:blueprint:config:)` — full-screen cover with a labelled Close button, fresh race session per presentation (`runID`). Used by TrackBuilder, Customizer, CharacterEditor and TestMode so the gesture — big yellow button, race, X, keep building — is identical everywhere.
  - `TryItButton` / `SaveItButton` — the paired workshop buttons ("See it in 3D!" / "Test Drive!" + "Save it!"). Both `lineLimit(1).fixedSize()`: in a crowded toolbar SwiftUI would rather stack a label into "Se / e it / in / 3D" than overflow, which reads as a broken button.

  Workshops preview what's **on screen**, not what's saved — they pass `AppModel.stampedRaceDesign(car:driver:)` overrides and (for the builder) the live `model.blueprint`. That's the whole point: no save, no backing out.
- `VFX.swift` — boost tailpipe glow (emissive material pulse + particle if cheap), debris despawn fade, confetti burst on win (`ParticleEmitterComponent`).

Perf budget (Apple TV 4K): ≤ 40 track pieces + 2 cars + debris ≤ 150 draw-relevant entities; static track merged where possible; target steady 60 fps. Profile with RealityKit debug options before adding VFX. If PiP reaction cam (2nd RealityView) costs > 10% frame time → sprite fallback per ReactionCam/README.
